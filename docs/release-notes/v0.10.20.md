## 🆕 新增功能

### L0: 全局并发 trial 信号量（buffered channel）

**解决**：阿里云 ECS 2C2G 被 5 个 trial 同时跑挂 OOM（2026-07-12 已观察到 1 次）。

**实装**：
- `backend/internal/courtroom/concurrency.go`（[L0 ConcurrencyLimiter](file:///d:/源码/FullStack/DecisionCourt/backend/internal/courtroom/concurrency.go)）—— buffered channel 信号量，max=5
- 接入 `service.withCancel` 入口（**L0 只在 service 层 1 处管理**，避免双重 acquire bug）
- wrappedCancel 在 defer cancel 时自动 Release
- 4 个 metric（rejected_total / current / max）

**面试关键**：
- 信号量 vs 限流的区别：信号量表达"资源硬上限"，限流表达"频率"
- 为什么选 buffered channel：Go 习惯写法 + select default 自然表达非阻塞

### L1: Per-Session action 限流（令牌桶）

**解决**：用户 F5 狂点同一 session 的"继续质证"按钮（实测 1 秒 30 次 action），把正常用户拖慢 30 倍。

**实装**：
- `backend/internal/middleware/session_ratelimit.go`（[L1 SessionRateLimit](file:///d:/源码/FullStack/DecisionCourt/backend/internal/middleware/session_ratelimit.go)）—— 复用 `golang.org/x/time/rate`，RPS=2 / Burst=5
- 按 session UUID 隔离（map[string]*rate.Limiter + sync.Mutex）
- 接入 `handler.RegisterAPIRoutes` 内 `llmGroup.Use(h.SessionRateLimit)`
- OnReject 回调接入 `MetricSessionRateLimitRejectedTotal`

**面试关键**：
- 选令牌桶 vs 漏桶：允许突发（F5 狂点 5 次连续仍通过）+ 平滑长期流量
- L1 接 HTTP middleware 层 vs L0 接 service 层：HTTP 层立即返 429（用户体验好），service 层走 WS 通道（业务级错误）

### L2: Per-User Trial 限流（滑动窗口，已有 v0.9 ADR 0014）

**未改动**，仅作为 4 层防御的第 2 层。

### L3: Per-IP 通用限流（令牌桶，已有 v0.8.3）

**未改动**，仅作为 4 层防御的第 3 层。

## 🐛 实施过程中的 2 个真实 bug（教训）

### Bug #1: handler 层 + service 层双重 acquire

**症状**：`TestService_WithConcurrencyLimiter_Limits` integration test fail —— 同 1 个 trial 占了 2 个 slot（max=5 实际只能开 2-3 个）。

**根因**：最初设计在 handler.StartTrial 入口直接 TryAcquire + service.withCancel 内部也 TryAcquire，同 1 个 trial 双 acquire。

**修复**：撤销 handler 层 TryAcquire，**L0 只在 service.withCancel 一处管理**。失败 UX 改为：HTTP 200 返回 → 后台 goroutine 失败 → WS 推送 `CodeConcurrentTrialLimit`。

**学习**：**限流器接入点选择 > 算法选择**。集成测试是唯一能暴露多层串扰的方式。

### Bug #2: Release 在空 channel 上阻塞 30s

**症状**：`TestConcurrencyLimiter_NoAcquireRelease_Panics`（预期 panic）实际 hang 30s 超时。

**根因**：`<-l.sem` 在空 channel 上是 receive 操作，**永久阻塞**。

**修复**：Release 改为 `select default` 非阻塞 + `slog.Warn`。生产环境优先避免进程卡死 30s。

**学习**：buffered channel 信号量的 Release 必须是宽容的（不阻塞 + log warning），不应 panic。

详见 [interview/12 §12.3](../interview/12-rate-limit-defense-in-depth.md) 完整 5 个教训。
